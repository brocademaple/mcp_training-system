# 基于 MCP 的模型训练 Multi-Agent 系统 · 2.0 产品规划

> 本文档从**资深产品经理**视角，对「1.0 手动 SaaS」升级为「2.0 Multi-Agent + MCP 驱动」的愿景、定位、架构与路线图做整体规划，便于评审、排期与落地。

---

## 一、项目定位与 1.0 / 2.0 边界

### 1.1 题目与目标

- **项目题目**：基于 MCP 的模型训练 Multi-Agent 系统  
- **1.0 现状**：  
  - 实现了从**数据处理 → 模型训练 → 模型评估**的完整业务链路，但以**手动操作**为主（用户逐页上传数据、创建训练、创建评估）。  
  - 后端虽有 DataAgent / TrainingAgent / EvaluationAgent 与 `internal/mcp` 下的 Coordinator、MCPMessage，但**主流程由 HTTP 请求直接调 Agent**，没有「以 MCP 消息为唯一驱动」的编排层，也没有对外暴露标准 MCP 协议。  
  - 因此 1.0 更准确的定义是：**具备 Agent 形态的、非 MCP 编排的手动 SaaS**。  

- **2.0 目标**：  
  - **Multi-Agent**：数据、训练、评估等能力以**独立 Agent** 存在，通过**统一的 MCP 消息**被调度与协同。  
  - **MCP 驱动**：核心业务流程（如「一份数据 → 清洗 → 训练 → 评估」）由 **MCP 消息流** 驱动，而不仅是 HTTP 直调；支持会话、上下文与可观测性。  
  - **可扩展**：为后续「外部 MCP 客户端接入」「多租户 / 多会话」留出清晰扩展点。

### 1.2 一句话对比

| 版本 | 一句话描述 |
|------|-------------|
| 1.0 | 用户在各页面手动完成「数据 → 训练 → 评估」，后端用 Agent 执行具体步骤，但无统一 MCP 编排与协议暴露。 |
| 2.0 | 用户或外部系统通过「目标 + 上下文」发起请求，由 MCP 编排层将目标拆解为 Agent 间的消息流，数据 / 训练 / 评估全链路由 MCP 驱动并可追溯。 |

### 1.3 双版本并存与互通（必达要求）

- **保留手动版**：当前 1.0 的**手动操作流程**作为独立模式长期保留（数据集管理、训练任务、模型管理、评估任务等页面及逐项创建能力均保留），与 2.0 Agent 版在同一系统内并存。
- **版本切换入口**：在系统内提供**版本切换**控件（如顶部导航或全局设置中的「经典版 / 手动版」⇄「Agent 版 / 智能编排版」），用户可随时切换当前使用的界面与工作流风格；切换后仍为同一套账号与数据。
- **数据互通**：  
  - 两版本**共用同一套数据**（同一 DB：datasets、training_jobs、models、evaluations 等）。  
  - 在 **Agent 版**中产生的数据变动（如新上传/清洗的数据集、由流水线产生的训练任务与模型、评估任务与报告）在 **1.0 手动版**中**可见、可用**：例如在手动版的数据集列表能看到 Agent 版创建的数据集，在模型列表中能看到 Agent 版流水线产出的模型，反之亦然。  
- **Skill / Workflow 在 1.0 可发现**：  
  - 在 Agent 版中沉淀的**技能（Skill）**、**工作流原型（Workflow 模板）**等，应在 1.0 版本中有**可发现、可追溯**的入口（例如「工作流模板」列表、或训练/评估列表中标注「由 Agent 版流水线创建」），以便用户在手动版中查看、复用或继续基于这些结果做手动操作。  
- **实现要点**：  
  - 版本切换仅影响**前端路由/展示与默认操作路径**（手动逐步操作 vs 一键流水线/会话视图），**不拆库、不拆表**；所有写入均落同一库表。  
  - 若 2.0 引入「工作流模板」「Skill 定义」等新实体，需有统一表或存储，并在 1.0 提供只读或跳转入口（如「在 Agent 版中查看 / 编辑」），保证两版能力互通。

## 二、2.0 核心概念（产品侧）

### 2.1 Multi-Agent 在本项目中的含义

- **Agent**：封装一类能力的执行单元，具备**输入（MCP 消息 Payload）、输出（结果 / 状态 / 错误）**，且可被**统一寻址与调度**（如 `data-agent`、`training-agent`、`evaluation-agent`）。  
- **Multi-Agent**：多个 Agent 在**同一套协议与编排规则**下协作；任务由**编排层**分解为多步，每步以 MCP 消息形式派发到对应 Agent，上一步输出可作为下一步输入的上下文。  

当前 1.0 已具备三个 Agent 的**能力边界**，2.0 的重点是：**用 MCP 消息明确「谁在什么上下文中执行什么动作」**，并让整条链路可观测、可复现。

### 2.2 MCP 在 2.0 中的角色

- **协议层**：  
  - 所有跨模块的调用（编排 → Agent、未来 Agent 间回调）均以 **MCP 消息**（如现有 `MCPMessage`：From / To / Action / Payload）为契约。  
  - 2.0 需补齐：消息 ID 关联、会话 ID、错误码与重试策略，以便追踪与排错。  

- **编排层**：  
  - 新增**编排器（Orchestrator）**（或强化现有 Coordinator）：接收「高层目标」（如「用数据集 5 训练并评估模型」），拆解为一系列 MCP 请求（clean_data → train → evaluate），按顺序或 DAG 派发，并维护**会话状态**（当前步骤、已完成步骤、中间结果 ID）。  

- **可观测性**：  
  - 所有 MCP 消息与关键状态变更可落库或写日志，便于 2.0 控制台/前端展示「当前流水线执行到哪一步、每步的输入输出摘要」；为后续 Agent 级监控与审计打基础。

### 2.3 用户价值（2.0 相比 1.0）

- **一键流水线**：用户选择「一份数据 + 训练配置」，系统自动完成「清洗 → 训练 → 评估」，并在一个视图里展示整条链路进度与结果。  
- **任务与模型强绑定**：在训练任务与模型管理视图中，明确展示「该任务产出的模型」「该模型对应的训练任务」，减少认知负担（与 1.0 方案 A 的「模型列 + 快捷评估」一致，2.0 可在此基础上增加 MCP 链路追溯）。  
- **可追溯与可复现**：每次流水线执行对应一个「会话」或「流水线实例」，保留 MCP 消息序列与关键结果 ID，便于排查与复现。  
- **为 3.0 留口**：外部 MCP 客户端（如 IDE、自动化脚本）可通过同一套 MCP 协议提交任务、查询状态，与前端共享同一套 Agent 与编排能力。

## 三、2.0 架构愿景（精简）

### 3.1 逻辑分层

```
┌─────────────────────────────────────────────────────────────────┐
│  接入层                                                          │
│  · 现有 Web 前端 + 版本切换（经典版/手动版 ⇄ Agent版/智能编排版）   │
│  · 两版共用同一 API 与数据源，仅路由与默认操作路径不同               │
│  · 未来：标准 MCP 服务端（stdio/SSE），供外部客户端连接              │
└─────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────┐
│  编排层（Orchestrator / 强化 Coordinator）                        │
│  · 接收「目标」：如 run_pipeline(dataset_id, train_config)          │
│  · 拆解为 MCP 消息序列：clean_data → train → evaluate             │
│  · 维护会话 / 流水线实例状态，落库或 Redis                          │
│  · 将「当前步骤、进度、错误」暴露给接入层与可观测性                   │
└─────────────────────────────────────────────────────────────────┘
                                    │
                    ┌───────────────┼───────────────┐
                    ▼               ▼               ▼
┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│ data-agent   │  │training-agent│  │evaluation-   │
│ (清洗/分析等) │  │ (训练/进度)   │  │ agent(评估)   │
└──────────────┘  └──────────────┘  └──────────────┘
                    │
                    ▼
┌─────────────────────────────────────────────────────────────────┐
│  执行层：Python 脚本 + DB + Redis（与 1.0 一致）                    │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2 与 1.0 的衔接

- **保留**：现有 HTTP 路由、三个 Agent 的实现、前端页面结构、DB 与 Redis 使用方式；**手动版全部能力保留**，与 Agent 版通过版本切换共存。  
- **新增/强化**：  
  - **版本切换**：前端提供版本切换按钮/入口，切换后进入「经典版」（当前手动流程）或「Agent 版」（一键流水线 + 会话视图）；后端不区分版本，仅提供两套路由或同一套 API 的不同使用方式。  
  - **统一数据**：datasets / training_jobs / models / evaluations 等表两版共用；Agent 版产生的数据在手动版列表与详情中可直接看到并操作（如用该数据集创建训练、用该模型创建评估）。  
  - **Skill / Workflow 可发现**：2.0 的流水线模板、Skill 定义等若落库，需在 1.0 提供入口（如「工作流与技能」页或列表中的来源标注），保证在手动版也能找到 Agent 版沉淀的能力。  
  - **MCP 消息驱动**：编排层只通过 MCP 消息调用 Agent；Handler 负责「创建流水线实例」「查询状态」等，具体步骤由编排层发 MCP 完成。  
  - **流水线实例与状态**：新增表或 Redis 结构，记录「流水线实例 ID、步骤列表、当前步骤、每步的 MCP 消息 ID 与结果摘要」，供前端「一键流水线」页与 2.0 可观测性使用；这些记录在手动版可通过「由 Agent 版创建」的维度查询或跳转。  
  - **可选**：实现标准 MCP 服务端（如 stdio 或 SSE），使外部工具能以 MCP 客户端身份发起「运行流水线」「查询状态」等，与 Web 共享同一编排与 Agent。

## 四、2.0 功能规划（按阶段）

### 4.1 阶段 2.0a：MCP 驱动 + 一键流水线（推荐优先）

- **目标**：在不破坏现有手动操作的前提下，增加「由 MCP 编排的一条龙流水线」，并让「任务 ↔ 模型」在 UI 上强绑定。  

- **产品能力**：  
  - 用户在一个界面选择：**数据集 + 训练配置（或使用默认）**，点击「一键：清洗 → 训练 → 评估」。  
  - 系统在后台按顺序通过 **MCP 消息** 执行：clean_data(dataset_id) → train(job_id) → evaluate(model_id, test_dataset_id)；每步的输入来自上一步的输出（如 job_id、model_id 由编排层从 DB 或 Payload 中获取）。  
  - 前端展示「流水线进度」：当前步骤（清洗中 / 训练中 / 评估中）、每步的简要状态或错误；完成后汇总「训练任务 ID、模型 ID、评估报告」并支持跳转。  

- **技术要点**：  
  - 完善 `Coordinator.ExecuteWorkflow`（或重命名为 `RunPipeline`）：在 clean_data 成功后创建训练任务并调用 train，训练成功后取 model_id 再调用 evaluate；每步以 MCP 消息形式经 Coordinator 派发。  
  - 新增「流水线实例」存储（表或 Redis）：pipeline_instance_id、steps、current_step、created_at、status；可选存储每步的 MCP message_id 便于追溯。  
  - **版本切换**：前端增加全局「版本切换」按钮（如 Layout 顶栏或侧栏底部），支持「经典版 / 手动版」与「Agent 版」切换；切换后路由到对应首页或工作台，两版共用同一 API 与 DB。  
  - 前端新增「一键流水线」入口与进度页（Agent 版）；训练任务列表保留并增强「模型」列与「创建评估」快捷入口（与 1.0 方案 A 一致）。手动版保留现有数据集 / 训练 / 模型 / 评估各页不变。  

- **验收**：  
  - 用户选择数据集并点击「一键流水线」后，无需再手动创建训练与评估，整条链路自动跑通并在同一视图看到进度与结果。  
  - 所有步骤在服务端通过 MCP 消息触发，可在日志或简单控制台看到消息流。

### 4.2 阶段 2.0b：会话与上下文 + 可观测性

- **目标**：为每次「用户意图」或「外部请求」赋予**会话 ID**，整条 MCP 消息链归属同一会话，便于追溯与排查。  

- **产品能力**：  
  - 流水线实例与「会话」绑定；支持在管理后台或「我的流水线」页按会话/时间筛选，查看某次执行的完整步骤与每步的输入输出摘要。  
  - Agent 在执行时可将「会话 ID」写入日志或 DB，便于用会话 ID 串联日志。  

- **技术要点**：  
  - MCPMessage 扩展：增加 optional `session_id`、`pipeline_instance_id`。  
  - 编排层在创建流水线时生成 session_id，并在后续所有派发的 MCP 消息中携带。  
  - 存储层：流水线实例表增加 session_id；可选「步骤日志」表（step_id, pipeline_instance_id, agent, action, payload_summary, result_summary, status, created_at）。  

- **验收**：  
  - 给定一次「一键流水线」执行，能在后台或界面按会话查看完整 MCP 步骤列表及每步的概要信息。

### 4.3 阶段 2.0c：标准 MCP 服务端 + 开放能力（可选）

- **目标**：对外暴露**标准 MCP 协议**（如 MCP over stdio 或 SSE），使 IDE、脚本等 MCP 客户端能连接本系统，提交任务与查询状态。  

- **产品能力**：  
  - 提供 MCP Server 端点，支持例如：`tools/list`、`tools/call`（如 `run_pipeline`、`get_pipeline_status`、`list_models` 等）。  
  - 外部客户端通过标准 MCP 调用这些 tool，与 Web 前端共享同一套编排与 Agent；权限与多租户可在此阶段引入（如 API Key、user_id 关联会话）。  

- **技术要点**：  
  - 引入官方或社区 MCP Go SDK，实现 MCP Server；将「运行流水线」「查询状态」等封装为 MCP tools。  
  - 内部仍通过现有 Coordinator + MCP 消息与 Agent 通信，MCP Server 仅作为「接入层」将外部调用转为内部消息或 HTTP。  

- **验收**：  
  - 使用独立 MCP 客户端（如 CLI 或 IDE 插件）能连接本系统并成功调用 run_pipeline、get_pipeline_status 等。

## 五、非功能与约束

- **双版本并存**：手动版（1.0 风格）与 Agent 版（2.0 风格）长期并存，通过**版本切换按钮**切换；不删除、不弱化现有手动流程。  
- **数据与能力互通**：两版共用同一数据源；Agent 版产生的数据、Skill、工作流原型在 1.0 手动版中可被发现与使用（同一列表、同一实体，或 1.0 提供「工作流/技能」查看入口）。  
- **兼容 1.0**：2.0a 上线后，现有「手动创建训练 / 手动创建评估」仍保留，与「一键流水线」并存；仅新增接口与前端入口，不删除旧能力。  
- **性能与资源**：编排层为轻量逻辑，主要耗时仍在 Python 训练与评估；需保留现有队列与并发策略，避免单机同时跑过多训练。  
- **安全**：若 2.0c 暴露 MCP 对外，需考虑认证、限流与审计（可沿用现有 user_id 或 API Key）。

## 六、总结与建议排期

| 阶段   | 主题                     | 建议优先级 | 产出物概要 |
|--------|--------------------------|------------|-------------|
| 2.0a   | MCP 驱动 + 一键流水线    | P0         | 编排层完整跑通「清洗→训练→评估」；前端一键流水线页与进度；**版本切换按钮**（经典版 ⇄ Agent 版）；两版共用数据，Agent 版结果在手动版可见。 |
| 2.0b   | 会话与可观测性           | P1         | 会话 ID 贯穿 MCP；流水线/步骤可查；**Skill / Workflow 原型在 1.0 可发现**（如工作流模板列表或来源标注）。 |
| 2.0c   | 标准 MCP 服务端与开放    | P2         | 外部 MCP 客户端可连接并调用 run_pipeline 等，与 Web 共享能力。 |

建议优先落地 **2.0a**，在现有代码基础上补齐编排层与前端「一键流水线」、**版本切换**及**数据互通**（同一 DB、两版同源列表）；随后再迭代 2.0b（可观测性 + 1.0 侧 Skill/Workflow 可发现）与 2.0c（开放协议）。这样既能快速体现「Multi-Agent + MCP 驱动」的题目内涵，又满足「保留手动版、双版互通」的必达要求。

## 七、更 Agent 化的产品建议（资深 PM 视角）

当前 Agent 版仍是「选数据集 → 点启动」的表单式操作，**意图感、拟人协作感、主动建议**不足。以下建议按优先级给出，便于在 2.0a/b 之上逐步增强「Agent 感」。

### 7.1 目标/意图优先，减少表单（P0，强烈建议）

- **现状**：用户需要自己选数据集、点「启动流水线」，本质还是填表。
- **建议**：  
  - 增加**预设目标**入口：如「从数据到评估」「只训练」「只评估已有模型」，选完再带出对应配置（默认数据集/模型可自动推荐或必选一项）。  
  - 或提供**一句话目标**输入框（如「用 ChnSentiCorp 训练情感分类并评估」），由系统解析出数据集名/ID 或给出候选，用户确认后开跑。  
- **价值**：从「我填表、系统执行」变为「我说目标、Agent 拆解并执行」，更符合 Agent 心智。

### 7.2 Agent 拟人化与状态可见（P0）

- **现状**：进度条 + 步骤名（数据清洗 / 模型训练 / 模型评估），看不出「谁在干活」。  
- **建议**：  
  - 执行中明确展示 **Data Agent / Training Agent / Evaluation Agent** 的名称与当前动作（如「Data Agent：清洗完成，已交给 Training Agent」）。  
  - 可选：简短 **MCP 消息流**时间线（可折叠），展示「谁发给谁、什么 action」，让多智能体协作可感知。  
- **价值**：用户能直观感受到「多个 Agent 在协作」，而不是一个黑盒进度条。

### 7.3 主动建议与「下一步」（P1）

- **现状**：流水线结束后只有「返回配置」，没有后续引导。  
- **建议**：  
  - 流水线**成功**：主动提示「要查看评估报告吗？」「要用同一模型换测试集再评估吗？」并给一键跳转或快捷操作。  
  - **上传/清洗完成**（在数据集页或 Agent 画布）：提示「新数据集已就绪，要加入流水线吗？」可预选该数据集并跳转到画布。  
- **价值**：从「被动等用户点」变为「Agent 主动建议下一步」，更智能。

### 7.4 失败时的 Agent 式修复建议（P1）

- **现状**：评估/训练失败只展示错误信息，用户需自己琢磨怎么改。  
- **建议**：  
  - 根据失败类型给出 **1～2 条可点击的「建议操作」**（如「改用训练集划分的测试集」「重试清洗」「检查测试集是否含文本/标签列」），点击后跳转到对应页面或触发对应接口。  
  - 可与现有「失败原因洞察」结合，把 `suggestions` 变成前端可执行的快捷操作。  
- **价值**：从「报错」变为「Agent 帮你指路」，降低排查成本。

### 7.5 技能/工具可见（P1）

- **现状**：画布上只有「选择数据集 + 启动流水线」，能力边界不清晰。  
- **建议**：  
  - 在画布展示**可用技能**（如「数据清洗」「模型训练」「模型评估」），并标注由哪个 Agent 提供（Data / Training / Evaluation）。  
  - 提供「从技能开始」的入口：例如点「评估」→ 选模型 + 测试集 → 只跑评估；或「只训练」→ 选数据集 → 只跑清洗+训练。  
- **价值**：用户理解系统能力边界，并能按需组合技能，而不是只能跑整条流水线。

### 7.6 会话与轻量记忆（P2）

- **现状**：每次进入画布都是「新会话」，没有对历史的利用。  
- **建议**：  
  - 同一会话内**记住最近使用的数据集/模型**，下次进入时默认推荐或置顶。  
  - 或提供「**沿用上次流水线配置**」一键再来一次（相同数据集、相同目标）。  
- **价值**：减少重复选择，体验更连贯。

### 7.7 统一入口：任务/目标框（P2）

- **现状**：画布以「选择数据集」为主入口，目标感弱。  
- **建议**：  
  - 将主入口改为**任务/目标**（预设目标选择 + 或 一句话描述），下方「高级选项」折叠（选具体数据集、模型、测试集等），默认用户只选目标即可跑通。  
- **价值**：界面更聚焦「我要做什么」，而不是「我要选什么」，更 Agent。

## 九、技术升级路线（微调与 Manager）

> 原独立文档 `UPGRADE_PLAN.md` 已合并至本节。

### 9.1 补全微调功能（最优先）

#### 目标

- 在现有「从零训练分类头」之外，增加 **SFT + LoRA 参数高效微调** 能力。
- 与现有训练链路共用：同一套 Go 训练任务接口、同一套进度/日志协议、同一套模型落库与评估流程。
- 改动量尽量小：仅新增一个 Python 脚本 + 后端按 `model_type` 路由。

#### 实现要点

| 层级 | 改动 |
|------|------|
| **Python** | 在 `python_scripts/training/` 下新增 `finetune.py`，使用 **TRL `SFTTrainer` + PEFT `LoraConfig`**，与 `train_text_clf.py` 保持相同 CLI 与 stdout 协议（`LOG:` / `PROGRESS:` / 最终 JSON）。 |
| **Go 后端** | `internal/agents/training_agent.go`：根据任务表中的 `model_type` 选择脚本——`sft_finetune` → `finetune.py`，其余（如 `text_classification`、`random_forest`）→ `train_text_clf.py`。 |
| **前端 / 流水线** | 创建训练任务或启动流水线时，在 `train_config` 中传入 `model_type: "sft_finetune"` 即可走微调脚本；无需改表结构。 |

#### 脚本接口约定（与现有一致）

- **入参**：`sys.argv[1]` = 数据集文件路径，`sys.argv[2]` = hyperparams 的 JSON 字符串（含 `job_id`、`base_model`、`epochs`、`learning_rate`、`batch_size` 等）。
- **stdout 协议**：过程日志 `LOG:`；进度 `PROGRESS:{...}`；结束单行 JSON `success` / `error`。

#### 依赖与触发

- `python_scripts/requirements.txt`：`trl[peft]`、`peft`（可选 `bitsandbytes`）；底模由 `train_config.base_model` 等传入。
- **触发**：`train_config.model_type: "sft_finetune"`；前端可增加「训练方式」选项对应 `text_classification` / `sft_finetune`。

#### 验收

- 任务可排队执行，进度与日志可见，结束后模型落库并可评估。

### 9.2 2.0 Multi-Agent 升级（smolagents CodeAgent 作为 Manager）

#### 目标

- 用 **smolagents 的 CodeAgent** 作为顶层 Manager，将 **Data / Training / Evaluation** 三个 Agent 暴露为 **MCP Tools**。
- 用户一句话描述目标，由 Manager 自动拆解、编排顺序与参数。

#### 思路

- 将「清洗」「训练」「评估」封装为 MCP Server 的 Tools；Python 侧用 `CodeAgent` / `ToolCallingAgent` 注册工具，由大模型（如 `Qwen/Qwen2.5-Coder-32B-Instruct`）决策调用顺序。
- 需为 Go MCP 定义 Tool schema；Manager 经 MCP 客户端调用。

#### 与 1.0 的关系

- 手动流程与 Agent 画布固定流水线保留；「一句话 + Manager」为新增入口，数据与表结构共用。

#### 阶段建议

1. **Phase 1**：微调链路可用（`finetune.py` + `sft_finetune` 路由）。  
2. **Phase 2**：Go 侧暴露标准 MCP Tools，Coordinator 可被 Tool 触发。  
3. **Phase 3**：引入 smolagents Manager，端到端 demo。

### 9.3 小结

| 项目 | 内容 |
|------|------|
| **微调** | `finetune.py` + Go 按 `model_type` 路由；与 `train_text_clf.py` 协议一致。 |
| **Multi-Agent** | 三 Agent → MCP Tools + CodeAgent Manager；分阶段落地。 |

相关实现：`python_scripts/training/finetune.py`、`internal/agents/training_agent.go`。
